Just a terminal.

  • Using a game engine teaches you the engine
  • Building from scratch teaches you the fundamentals
  • Architecture patterns are clearer without engine abstractions hiding them

Procedural Generation

Maze generation in pure Ruby

class BinaryTree
  def self.on(grid)
    grid.each_cell do |cell|
      neighbors = []
      neighbors << cell.north if cell.north
      neighbors << cell.east  if cell.east
      neighbor = neighbors.sample

      cell.link(cell: neighbor) if neighbor
    end
  end
end

For each cell, randomly link north or east. Every run produces a unique maze.
Four algorithms in total — each teaches different principles about graph theory and spanning trees.

It Worked... Until It Didn't

The naive Game class

# Everything in one place. Input, movement, combat, rendering.
def check_combat
  @monsters.each do |monster|
    if monster.x == @player.x && monster.y == @player.y
      damage = @player.attack_power - monster.defense
      monster.health -= damage
      if monster.health <= 0
        @monsters.delete(monster)
        @player.gold += monster.gold_drop
      end
    end
  end
end

This code breaks the moment you want: flying monsters, damage effects,
combat events, items that fight back, or any new feature.

The Problem

Why inheritance doesn't work

OOP inheritance

Player: Combat + Movement + Inventory
Monster: Combat + Movement + AI

Flying monster? Flying item? Item that fights back?

Exponential class explosion.

Composition (ECS)

Entity + PositionComponent + CombatComponent = fighter

Add FlyingComponent = flying fighter
Remove CombatComponent = peaceful

Any combination. No inheritance.

The Solution

Entity-Component-System

🆔
Entities
Just IDs. Containers for components. No behaviour, no data of their own.
📦
Components
Pure data. PositionComponent, HealthComponent, InventoryComponent. No logic.
⚙️
Systems
Logic that operates on entities with specific component combinations.

"What components does this entity have?" replaces "What class is this?"

In Practice

Systems query for what they need

class MovementSystem < System
  def process
    world.query(:position, :movement, :input).each do |entity|
      pos   = entity.get(:position)
      move  = entity.get(:movement)
      input = entity.get(:input)

      target = pos.cell.neighbors[input.direction]

      if target && pos.cell.linked?(target)
        pos.cell = target
        emit(:entity_moved, entity: entity, from: pos, to: target)
      end
    end
  end
end

MovementSystem doesn't know about combat. Doesn't know about rendering.
It moves things. That's it.

Communication

Event-driven architecture

  • Without events: System A directly calls System B → tight coupling
  • With events: System A emits :combat_damage, whoever cares subscribes
  • Every action generates events logged to event_logs/*.jsonl
  • Complete audit trail of every game state change
  • Debugging = reading the event log, not stepping through code

Logging, debugging, replay, analytics — all come free
when your architecture is event-driven.

Takeaways
