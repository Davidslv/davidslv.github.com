Procedural generation

Maze generation in pure Ruby.

class BinaryTree
  def self.on(grid)
    grid.each_cell do |cell|
      neighbors = []
      neighbors << cell.north if cell.north
      neighbors << cell.east  if cell.east

      neighbor = neighbors.sample
      cell.link(cell: neighbor) if neighbor
    end
  end
end

For each cell, randomly link north or east — every run a unique maze. Four algorithms in all, each a lesson in graph theory and spanning trees.

# everything in one place: input, movement, combat, rendering
def check_combat
  @monsters.each do |monster|
    if monster.x == @player.x && monster.y == @player.y
      damage = @player.attack_power - monster.defense
      monster.health -= damage
      if monster.health <= 0
        @monsters.delete(monster)
        @player.gold += monster.gold_drop
      end
    end
  end
end

This breaks the moment you want flying monsters, damage effects, combat events, or items that fight back. Every new feature touches it.

The problem · why inheritance doesn’t scale
OOP inheritance
Player: Combat + Movement + Inventory
Monster: Combat + Movement + AI

A flying monster? A flying item? An item that fights back?

Exponential class explosion.
Composition · ECS
Entity + Position + Combat = a fighter

Add Flying → a flying fighter
Remove Combat → peaceful

Any combination. No inheritance.
The solution · Entity–Component–System
🆔
Entities
Just IDs — containers for components. No behaviour, no data of their own.
📦
Components
Pure data, no logic — Position, Health, Inventory, Combat.
⚙️
Systems
All the logic — each operates only on entities with the components it needs.

“What components does this entity have?” replaces “What class is this?”

In practice · systems query for what they need
class MovementSystem < System
  def process
    world.query(:position, :movement, :input).each do |entity|
      pos   = entity.get(:position)
      input = entity.get(:input)
      target = pos.cell.neighbors[input.direction]

      if target && pos.cell.linked?(target)
        pos.cell = target
        emit(:entity_moved, entity: entity, to: target)
      end
    end
  end
end

It doesn’t know about combat. Doesn’t know about rendering. It moves things — and emits :entity_moved. That’s it.

Communication · event-driven architecture
withoutSystem A calls System B directly → tight coupling
withSystem A emits :combat_damage — whoever cares subscribes
loggedevery action appended to event_logs/*.jsonl — a complete audit trail
debuggingreading the event log, not stepping through code

Logging, debugging, replay, analytics — all come free when the architecture is event-driven. You saw every one of those events scroll past in the demo.
