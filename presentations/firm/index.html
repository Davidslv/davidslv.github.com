What it is

A firm of specialists you run
on your own repository.

Nineteen skills and five agents, each owning one seam of the engineering process — orchestrated by a producer that keeps them coherent. It handles the work around the code. It is the meta-tool; your repo is the product.

One engineer wearing every hat —
with a producer keeping
them coherent.
The organising idea · the seam

A seam is one standard,
owned by one role,
respected by all.

The reviewer who owns data safety sets that bar; no one else may lower it. One owner per seam, no overlap — and a gate (check-roster.sh) enforces it mechanically, so the contract can't quietly rot.

A reviewer that guesses your repo
is an opinion. One that reads its

The Profile is data, not prose — engines, asset layout, test command, integrations, criticality. The same reviewer grounds itself against whatever repo it's pointed at.

The evaluations · does the skill actually help?

Every judgment skill is run
with it and without it — and judged.

WITH / WITHOUTthe same task, the skill on vs the bare model — does the skill add anything?
3-VOTE JUDGEeach case scored by a 3-judge majority, not a single coin-flip
DELTA+ beat the baseline · 0 matched it · − regressed. A skill that only ties hasn't earned its slot
FORCING FUNCTIONSskills whose value is that they run every cycle are proven by a gate, not lift — a one-shot eval can't measure them
Verified 3-vote · 2026-06-20 · the honest picture
understatedsingle-judge lows were mostly too harsh — security-reviewer 2/3 → 4/4, communicator 1/3 → 3/4, engine-boundary 2/4 → 4/4.
heldclean 3/3, genuinely — story-writer, architecture-advisor, test-strategist, observability-check.
fellthe skeptic fabricates findings (1/3); perf-reviewer grades a real FK-index catch "likely", not "verified" (3/3 → 2/3).
why publish ita roster that only lists its wins is marketing. The deltas that go down are the ones worth trusting.
A design commitment · firm leaves no trace

It works on your repo
without writing into it.

The Profile and flag ledger live under a gitignored .firm/ — regenerated on demand, never committed by default. Seeding the repo's own knowledge files is an explicit opt-in. Run it on any repo; leave the git history exactly as you found it.

The honest part · firm is not a miracle

It's only as good as
what you ground it on.

Garbage in, faster garbage out
no runbooks, no structured docs → confident, wrong, at speed
Long-lived codebases lie

Tooling is leverage
give it access — tests, scanners, arsenal, ux-auditor — and it gets sharper
Feeding it well is the work

I know what you're about to ask
vs a checklist?A checklist is a reminder you can ignore. A seam has an owner and a gate — the build fails when it slips, and the loop sharpens it.
"The LLM already knows this"?It does — and skips it under pressure exactly like a person. The evals exist to show where the skill beats the bare model, and where it doesn't.
"Just more process"?The producer fires only the stations a change actually crosses. A typo gets nothing; a migration gets the full rigour.
vs my CI?CI checks the code. firm checks the work around the code — the design, the decisions, the decomposition, the lesson — and feeds your CI better inputs.
The honest limits

Core is universal; the deep reviewers are Ruby / Rails for now. Some skills still lose their eval — and I left them in the roster losing. And it needs grounding: docs, runbooks, tooling.

It routes, enforces, and remembers — you still do the engineering.

Is firm itself public?Not yet — Davidslv/firm ships once it's had more testing in the wild. Happy to ping you when it does.
LLM judging an LLM — circular?3-vote majority for the judgments; deterministic gates exist because judge-eval can't prove everything.
Why ship the skeptic if it loses?Publishing the losers is the credibility. Flagged, open follow-up. Only-wins is marketing.

Haven't you just moved the hard problem?Yes — firm makes the absence of docs expensive and visible. Often the first reason a team fixes it.
