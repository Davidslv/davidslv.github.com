The Problem

Rails apps that start beautifully
become terrifying at scale

app/models/ with 200 files. A User model with associations to everything. Service objects in app/services/, app/interactors/, app/queries/, app/decorators/.

Each new directory is a well-intentioned attempt to manage complexity.
None of them create actual boundaries.

The Cost

The cost of change becomes proportional
to the size of the entire application

Without boundaries

$ bin/rails test test/models/billing/invoice_test.rb
# 3 tests, 3 assertions, 0 failures ✓

$ bin/rails test
# 847 tests, 1203 assertions, 4 failures ✗
# Failures in: notification_mailer_test.rb,
# report_generator_test.rb, admin_dashboard_test.rb,
# webhook_handler_test.rb

With engine boundaries

$ cd engines/billing && bundle exec rspec
# 94 examples, 0 failures ✓

$ cd ../.. && bundle exec rspec
# 312 examples, 0 failures ✓

# Zero collateral damage.

Conway's Law

Your code structure mirrors
your team structure

Single team, no boundaries

One team → one monolith. Everyone works on everything. Good luck coordinating.

Teams aligned to engines

Team Billing owns engines/billing/. Team Notifications owns engines/notifications/. Changes don't require cross-team coordination.

"Any organisation that designs a system will produce a design whose structure is a copy of the organisation's communication structure." — Melvin Conway, 1968

The Mechanism

Rails Engines: built-in since 3.1

Mountable, namespace-isolated, independently testable modules
that plug into a host application.

  • The same mechanism that powers Devise, Spree, and ActiveAdmin
  • Not a gem from a startup. A core framework feature
  • isolate_namespace enforces boundaries at every layer: models, routes, views, tables
  • Each engine has its own test suite via a dummy app

Architecture

A modular Rails application

orbit/
├── app/                  # Host application (thin orchestration layer)
├── engines/
│   ├── core/             # Shared kernel — cross-cutting concerns
│   ├── billing/          # Invoices, subscriptions, plans
│   └── notifications/    # In-app notifications
└── config/               # Host configuration wires engines together

The host app is thin. Each engine owns its domain.
Engines communicate through events and concerns, not direct references.

Bounded Contexts

The User model as a composition point

class User < ApplicationRecord
  include Auth::Authenticatable      # password, sessions, tokens
  include Billing::Billable          # subscriptions, invoices
  include Notifications::Notifiable  # preferences, device tokens
  include Teams::Membership          # memberships, roles
end

Each line is an explicit declaration: "User participates in this domain."
Remove a line, and the user loses that capability. The boundaries are visible in the code.

Finding Boundaries

Your git history already knows
where the boundaries are

# Files that frequently change together = same domain
  47 times: app/models/invoice.rb
           app/services/invoice_generator.rb

  38 times: app/models/invoice.rb
           app/models/subscription.rb

   3 times: app/models/invoice.rb
           app/models/notification.rb
           # ← Weak coupling. Boundary signal.

Run a co-change analysis. The clusters are your candidate engines.
The sparse connections are your integration points.

Integration

How engines communicate

  • Concerns — engines expose, hosts include (Billable, Notifiable)
  • Events — ActiveSupport::Notifications for cross-engine communication
  • Configuration — dependency inversion via mattr_accessor
  • Shared kernel — engines/core/ for cross-cutting concerns (Auditable)
  • Background jobs — event subscribers enqueue jobs, never process inline

No engine reaches into another engine's internals.
The integration points are explicit and testable.

Not Microservices

The same operation.
Different complexity.

Modular monolith

15 lines. One transaction. If anything fails, everything rolls back.

Microservices

40+ lines. Two HTTP calls. Two failure modes requiring compensating transactions. A third with no clean recovery.

The code complexity difference is not 2× — it's an order of magnitude,
and the failure modes multiply.

Honest Trade-offs

When engines are the wrong tool

  • < 20 models — namespaces and plain Rails are enough
  • < 6 developers — you don't have a coordination problem
  • Zero cross-domain test failures last quarter — the coupling cost is low
  • Test suite under 5 minutes — the testing benefit is marginal
  • Boundaries aren't clear yet — use Packwerk to analyse first

Being honest about when your tool isn't the right one
is a sign of maturity, not weakness.

The Framework

Four questions, in order

1
Is a single Rails app sufficient?
Yes → stop. Use namespaces.
2
Do domains need structural isolation?
Yes → use engines. Modular monolith.
3
Do domains need process-level isolation?
Yes → promote specific engines to services.
4
Do domains need different technology?
Yes → build separate services.

Most applications never get past step 2. And that's fine.
