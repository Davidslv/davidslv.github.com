The problem

Rails apps that start beautifully
become terrifying at scale.

app/models/ with 200 files. A User model with associations to everything. Every feature touches the same handful of god-objects.

Folders are not boundaries
# every team's well-meant attempt to manage the sprawl
app/services/      → still imports anything
app/interactors/   → still imports anything
app/queries/       → still imports anything
app/decorators/    → still imports anything
app/models/        → 200 files, all reachable from all

They organise the code. None of them create an actual boundary. A class in one folder can still reach straight into another.

A folder is a label.
A boundary is a wall
something enforces.

Conway’s law

Your code structure mirrors
your team structure.

single team · no boundaries
One team → one monolith. Everyone works on everything. A coordination tax on every change.
teams aligned to engines
Team Billing owns engines/billing/. Team Notifications owns engines/notifications/. Changes stop needing cross-team coordination.

“Any organisation that designs a system will produce a design whose structure is a copy of the organisation’s communication structure.” — Melvin Conway, 1968

The mechanism

Rails Engines: built in since 3.1.

Mountable, namespace-isolated, independently testable modules that plug into a host application.

  • The same mechanism that powers Devise, Spree and ActiveAdmin
  • Not a gem from a startup — a core framework feature
  • isolate_namespace enforces boundaries at every layer: models, routes, views, tables
  • Each engine carries its own test suite via a tiny dummy app
Architecture

A modular Rails application.

orbit/
├── app/                  # host app — thin orchestration layer
├── engines/
│   ├── core/             # shared kernel — cross-cutting concerns
│   ├── billing/          # invoices, subscriptions, plans
│   └── notifications/    # in-app notifications
└── config/               # host wires the engines together

The host is thin; each engine owns its domain. Engines communicate through events and concerns — never direct references.

The boundaries aren’t in a wiki.
They’re visible in the code.

Bounded contexts

The User model as a composition point.

class User < ApplicationRecord
  include Auth::Authenticatable      # password, sessions, tokens
  include Billing::Billable          # subscriptions, invoices
  include Notifications::Notifiable  # preferences, device tokens
  include Teams::Membership          # memberships, roles
end

Each line declares: “User participates in this domain.” Remove a line and the user loses that capability — cleanly. The boundaries are visible.

Integration · how engines communicate
concernsengines expose, hosts include — Billable, Notifiable
eventsActiveSupport::Notifications for cross-engine communication
configurationdependency inversion via mattr_accessor
shared kernelengines/core/ for cross-cutting concerns — Auditable
background jobsevent subscribers enqueue jobs, never process inline

No engine reaches into another engine’s internals. The integration points are explicit and testable.

orbit · engine test feedback loop
4 min
full feedback loop  ·  vs ~40 for the equivalent monolith
Each engine tests against a minimal dummy app — no notification system to load, no admin panel to configure. ~2s boot, tens of seconds per engine suite, and CI runs them in parallel.
Not microservices · the same operation
modular monolith
~15 lines. One transaction. If anything fails, everything rolls back. One failure mode.
microservices
40+ lines. Two HTTP calls. Two failure modes needing compensating transactions — and a third with no clean recovery.

The complexity difference isn’t 2× — it’s an order of magnitude, and the failure modes multiply.

Honest trade-offs · when engines are the wrong tool
< 20 models
namespaces and plain Rails are enough
< 6 developers
you don’t have a coordination problem
No cross-domain failures last quarter
the coupling cost is already low
Suite under 5 minutes
the testing benefit is marginal

Boundaries not clear yet? Analyse with Packwerk first. Being honest about when your tool is wrong is maturity, not weakness.

The framework · four questions, in order
1Is a single Rails app sufficient? Yes → stop. Use namespaces.
2Do domains need structural isolation? Yes → engines. Modular monolith.
3Do domains need process-level isolation? Yes → promote specific engines to services.
4Do domains need different technology? Yes → build separate services.

Most applications never get past step 2. And that’s fine.
