Curveball · failure / what you'd do differently

Tell me about a time you failed.

failedI jump to the solution before I've understood the problem — built things, incl. a full product, for the engineering, with no proven need
costsunset one that was costing money to run — the work was good, but no demand underneath it
learnedthe problem must be present and well understood before I build — first question now: proven, or just assumed?
proofso on a large speculative rebuild, I pushed to fix the existing system first — saved months building nothing we'd shown we needed
showsself-awareness · a closed loop (failed → learned → proof) · problem-first judgement
Follow-ups they'll push (4) — answer each aloud, then check
Give me the specific product.
Name it plainly — it was mine: a tax & investment calculator I built and later sunset once it was clearly costing money with no validated demand. own the shutdown
How do you validate a problem is real now?
Look for the pain before building: talk to the people who'd use it, find evidence they're already working around the gap, run the smallest possible test before committing real effort. systemic fix
When is it fine to build something speculative?
Cheap, time-boxed spikes are fine and useful. The failure mode is committing months to an unproven assumption — match the size of the investment to the strength of the evidence. name the trade-off
Doesn't that slow you down?
The opposite. Once the problem's proven I move fast and commit hard, because I'm not hedging. The caution sits only at deciding what to build, never how fast. reframe
Alternate failure: the over-batched upgrade (3) — pick ONE, don't carry both in
The story: I bundled too much into one big dependency & runtime-upgrade batch — a judgement error driven by approval friction. Something reached prod a smaller batch would've caught. Now I cut changes into the smallest risk-based slices, each with its own verification. It's why I care about a fast, trustworthy pipeline.
Why batch it instead of splitting?
Honestly, approval friction — I let the slowness of the process push me into a bigger-bang change. That's the instinct I corrected. give the shape
How do you decide batch size now?
By risk, not convenience — smallest shippable slice with its own verification, even if that's more rounds. systemic fix
What did it cost?
Caught risk in prod rather than before it — partly luck, and relying on luck is what I removed. systemic fix
Pick one the night before — problem-before-solution (above) or the over-batched upgrade. Don't carry both into the room.

STAR I · driving process change · ~90s

Tell me about a time you improved how the team works.

Sslow deploy pipeline + undocumented systems dragging the team
Acut the pipeline ~2h → 35 min · upgraded the runtime ahead of EOL · documentation + shared architectural vocabulary
Rfaster, safer delivery · a shared language for architecture decisions
showsplatform thinking · raising the team's floor
Follow-ups — derived, vet these (4) · press D
Why does deployment speed matter so much to you?
A slow pipeline pushes people toward big risky batches; fast, trustworthy delivery lets you ship small and safe. It's the lesson behind my failure story. reframe
How did the shared vocabulary actually get adopted?
I wrote it down and used it consistently in reviews and docs until it became the team's default — adoption by usefulness, not mandate. give the shape
How exactly did you cut the pipeline time?
The levers: cache the expensive dependency we were recompiling every build; Docker layer caching + a .dockerignore so we stopped rebuilding from scratch; parallelise the test suite (our PR CI already runs it in parallel); slim the images by dropping packages that didn't belong in production; and run the independent test and build stages in parallel instead of in series. give the shape
What did the parallelism cost?
More CI compute minutes — I traded a little spend for a lot of wall-clock, and quantified it so it was a conscious call. Production stays gated; only the independent, non-prod stages run concurrently. trade-off

Holding ground under probing · press M anytime to return here

Six moves for the second and third question.

1Answer, then ONE layer. Answer what's asked, add a single layer — keep depth in reserve, let them probe where they care.
2Decision + why first, not a timeline. "I chose X because Y" reads senior; a play-by-play reads junior.
3Name the trade-off = judgement. "I considered X, but chose Y because Z." The rejected alternative is the signal.
4End on the systemic fix. "And here's what I changed so it can't recur." (their "constantly seek improvement").
5Blameless + name the boundary. Never blame or name a person/partner. "I can't get into their internals, but the pattern was…"
6Don't know? Say so cleanly. Never invent a number — give the shape, say you'd check, bridge back to the message.
Technical depth · performance · the one they'll go 3 deep on

Walk me through a hard performance problem you fixed.

Sa core internal view timing out at ~65 minutes (504) — effectively unusable
Aprofiled it: an unbounded COUNT across a large table every load + missing indexes → countless pagination + composite indexes on the real access pattern
R65-min timeout → under a second
showsprofile, don't guess · a boring root cause beats a clever optimisation
Follow-ups they'll push (3) — answer each aloud, then check
How did you find the cause?
Profiled the slow request, didn't guess. The count dominated, and the query plan showed sequential scans where there should've been index lookups. give the shape
Why was the count so expensive?
The default pagination counts the whole filtered set on every load; on a large table that's the killer. A pagination that doesn't need the full count removes it entirely. depth
Why composite indexes, not single-column?
The query filtered AND ordered on a combination, so a composite matching it satisfies in one go vs intersecting several. If I blanked on exact column order I'd say I'd check — but the shape was one index on the filter-plus-sort columns. never bluff
